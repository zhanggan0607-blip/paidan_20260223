# 业务逻辑对比分析文档

## 一、用户描述的业务逻辑

根据用户描述，业务流程如下：

```
1. /project-info → 生成项目信息
         ↓
2. /maintenance-plan → 根据项目信息制定维保计划
         ↓
3. /work-plan → 显示生成的维保计划（定期巡检单）
```

**核心关系链**：项目信息 → 维保计划 → 定期巡检单

---

## 二、本地代码实现的业务逻辑

### 2.1 数据模型关系

```
ProjectInfo (项目信息)
    ├── MaintenancePlan (维保计划) - 一个项目可有多个维保计划
    │       ├── PeriodicInspection (定期巡检工单)
    │       ├── TemporaryRepair (临时维修工单)
    │       └── SpotWork (零星用工工单)
    │
    └── WorkPlan (工作计划) - 工单汇总视图表
```

### 2.2 实际业务流程

```
1. 项目信息 (/project-info)
   - API: /project-info
   - 功能: 创建/管理项目基本信息
   - 权限: 管理员或部门经理

         ↓

2. 维保计划 (/maintenance-plan)
   - API: /maintenance-plan
   - 功能: 基于项目信息创建维保计划
   - **关键操作**: 创建维保计划时自动生成定期巡检工单
   - 权限: 管理员或部门经理

         ↓ (自动同步)

3. 定期巡检工单 (PeriodicInspection)
   - API: /periodic-inspection
   - 功能: 工单执行、填写、审批
   - 权限: 运维人员可执行，管理员可审批

         ↓ (同步到)

4. 工作计划 (/work-plan)
   - API: /work-plan
   - 功能: 统一显示所有工单（定期巡检+临时维修+零星用工）
   - 权限: 管理员可查看所有，运维人员只能查看自己的
```

| 模块 | 文件路径 | 功能说明 |

| 项目信息API | [backend-python/app/api/v1/project_info.py](backend-python/app/api/v1/project_info.py) | 项目信息CRUD接口 |
| 维保计划API | [backend-python/app/api/v1/maintenance_plan.py](backend-python/app/api/v1/maintenance_plan.py) | 维保计划CRUD接口 |

## 三、差异对比分析

### 3.1 概念差异

| 用户描述 | 本地代码实现 | 差异说明 |
|----------|--------------|----------|
| 维保计划 | MaintenancePlan | ✅ 一致 |
| 定期巡检单 | WorkPlan页面 | ⚠️ **不一致** - 本地WorkPlan是工单汇总视图，不是维保计划 |
| 维保计划生成定期巡检单 | 维保计划创建时自动生成PeriodicInspection | ✅ 逻辑一致，但名称不同 |

### 3.2 页面功能差异

#### 用户描述的 `/work-plan` 页面
- 显示"生成的维保计划"
- 即定期巡检单

#### 本地代码的 `/work-plan` 页面
- 显示**所有工单的汇总列表**
- 包含三种类型：定期巡检、临时维修、零星用工
- 是一个工单查询/管理页面，不是维保计划页面

### 3.3 数据流转差异

#### 用户期望的流程
```
项目信息 → 维保计划 → 定期巡检单 (work-plan显示)
```

#### 本地代码实际流程
```
项目信息 → 维保计划 → 自动创建定期巡检工单
                              ↓
                        同步到WorkPlan表
                              ↓
                    work-plan页面显示所有工单
```

## 四、核心差异总结

### 4.1 主要差异点

1. **页面定位不同**
   - 用户描述：`/work-plan` 显示"定期巡检单"
   - 本地实现：`/work-plan` 显示"所有工单汇总"（定期巡检+临时维修+零星用工）

2. **数据模型层级**
   - 本地代码中存在**WorkPlan**表，作为工单汇总视图
   - 定期巡检单实际存储在**PeriodicInspection**表
   - 维保计划存储在**MaintenancePlan**表

3. **自动生成逻辑**
   - 本地代码在创建维保计划时会**自动创建**定期巡检工单
   - 工单编号格式：`XJ-{项目编号}-{日期}`

### 4.2 是否满足用户需求？

| 需求 | 满足情况 | 说明 |
|------|----------|------|
| 项目信息生成 | ✅ 满足 | `/project-info` 页面可创建项目 |
| 根据项目制定维保计划 | ✅ 满足 | `/maintenance-plan` 页面可选择项目创建计划 |
| 维保计划生成定期巡检单 | ✅ 满足 | 创建维保计划时自动生成 |
| work-plan显示定期巡检单 | ⚠️ 部分满足 | 显示所有工单，包含定期巡检单，但不只是定期巡检单 |

1. `/work-plan` 页面的定位不同
