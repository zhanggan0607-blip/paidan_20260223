# 个人开发规范标准

> 按照"环境准备 → 问题定位 → 代码修改 → 验证记录 → 代码清洁 → 架构审查 → 安全审查 → 测试规范 → 部署与日志 → 最终收尾"的开发全流程编排，可直接作为后续所有工作的指导原则。

---

| 工具 | 镜像地址 | 说明 |

> 后续所有"构建验证"均指以上两条命令。

## 二、问题定位（先查后问）

### 2.1 必须先查阅 TROUBLESHOOTING.md

每次修复问题前，打开 `TROUBLESHOOTING.md`，确认是否为已知问题，是否有历史修复方案。

- 避免重复踩坑，复用已有经验
- 若文档前后有冲突，以最后日期的记录为准

### 2.2 不确定时提出假设，严禁猜测

遇到不确定的需求、模糊逻辑、歧义接口，**必须向用户提问，禁止靠猜测决定**。

可提出假设，但必须明确标注为假设，格式：

```
假设：XXX，因为 YYY
```

**验证假设的方式：**

1. 查阅项目已有代码的类似实现
2. 查阅 `TROUBLESHOOTING.md` 历史记录
3. 查阅后端/前端接口文档
4. 向用户提问确认

> ⚠️ 假设未验证前不能当作事实使用。

## 三、代码修改（最小代码量原则）

### 3.1 只解决当前问题

- **拒绝"屎山代码"**：不为"以后可能用到"写代码
- 每次修改只针对当前问题，不做额外重构、不加额外功能
- 如果发现其他问题，记录到待办，不在本次修改中一并处理

### 3.2 文件不超过 200 行

- 每个文件代码量**不得超过 200 行**
- 超过时必须拆分，优先提取为独立模块 / composable / service
- 拆分原则：按**职责单一性**，不按功能堆叠

### 3.3 只碰必要代码

- 修改范围严格限定在与问题直接相关的代码
- 禁止顺手"优化"无关代码，不调整无关格式、命名、注释
- 不引入不必要的依赖

## 四、验证与记录

### 4.1 修改后必须验证

验证方式优先级（从高到低）：

1. 运行项目已有单元测试 / 集成测试
2. 执行构建命令确认编译通过
3. 启动服务进行接口测试
4. 如无测试，手动构造验证场景

### 4.2 验证闭环

```
修改代码 → 运行验证 → 通过 → 完成
                  → 失败 → 分析原因（不盲目试错）→ 再次修改 → 重新验证
```

### 4.3 必须写入 TROUBLESHOOTING.md

每次修复或清理操作后，必须将信息记录到 `TROUBLESHOOTING.md`。

**记录格式：**

```markdown
## YYYY-MM-DD 修复/清理：简述问题

### 类型
修复 / 废弃删除 / 重复合并 / 共用提级

### 概要
一句话说明做了什么

### 核心根因（修复时填写）
1. 根因1

### 明细
- **文件**: 涉及文件列表
- **改动**: 具体改动内容（合并：合并前位置 → 合并后位置；提级：原位置 → 提级后位置）
```

### 4.4 复盘清单（每次修改后逐项确认）

- [ ] 原始问题是否完全解决？
- [ ] 是否只修改了必要代码？
- [ ] 修改后的文件是否超过 200 行？
- [ ] 是否已写入 TROUBLESHOOTING.md？
- [ ] 是否已通过验证（构建/测试）？
- [ ] 是否引入了新问题？
- [ ] 是否有遗漏的关联模块需要同步修改？
- [ ] 是否按代码清洁规则进行过检查？（见第五章）

## 五、代码清洁与统一管理（日常审查）

### 5.1 三项核心原则

| 原则 | 操作流程 |
|------|----------|
| **废弃即删** | 全局搜索无引用 → 构建验证 → 确认删除 → 记录到 TROUBLESHOOTING.md |
| **重复即合并** | 同一逻辑多处定义必须合并；前端常量/枚举/工具/类型统一放入 `src/constants/`、`src/utils/`、`src/types/`；后端对应放入 `constants/`、`enums/`、`util/` 包 |
| **共用即提级** | 被 2 个及以上模块引用的内容，必须提级到公共层，原位置只保留引用 |

### 5.2 审查范围（19 类检查项）

每次修改涉及以下范围时，必须同步检查是否存在废弃/重复/可共用内容：

| 类别 | 检查项 |
|------|--------|
| 文件/目录 | 废弃文件、空目录、未被引用的模块文件 |
| 依赖 | 未使用的依赖、重复依赖、版本冲突 |
| 环境 | 未使用的环境变量、多环境配置不一致 |
| API路径 | 前后端不匹配的接口 |
| 路由 | 指向不存在组件的路由、重复路由 |
| 组件 | 未引用的组件、功能重叠的组件 |
| 函数/方法 | 未调用的函数、功能相同名称不同的函数 |
| 常量/枚举 | 未使用的枚举值、前后端重复定义 |
| 类型定义 | 未使用的类型/接口、与后端不一致的前端类型 |
| 数据库表 | 无代码引用的表、无业务含义的中间表 |
| 数据库字段 | 实体有但 DTO 未映射且无查询使用的字段 |
| DTO | 未被 Controller 使用的 DTO、字段与实体不对应的 DTO |
| 注解 | 无实际效果的注解、重复注解 |
| 配置项 | 未使用的配置、重复配置 |
| 权限定义 | 代码未校验的权限、硬编码未录入权限表的权限 |
| 样式 | 未引用的 CSS class、重复样式 |
| 静态资源 | 未引用的图片/字体/图标 |
| 中间件/拦截器 | 无实际逻辑的拦截器、功能重叠的中间件 |
| 引用(import) | 导入未使用的模块、可合并的引用路径 |

### 5.3 操作规范

- 删除/合并/提级前，必须**全局搜索确认无引用**
- 每次清理后执行**构建验证**
- 清理操作按 [4.3 格式](#43-必须写入-troubleshootingmd) 记录

## 六、全面架构审查与"屎山"清理

> **触发时机**：用户明确要求、大规模重构、或累计修复超过 10 项时建议执行。

### 6.1 审查内容（整合架构、可提级、屎山、废弃冗余）

涵盖以下维度：

| 维度 | 检查要点 |
|------|----------|
| 代码质量 | 代码规范、重复代码、注释完整性、命名规范、屎山代码（过度耦合、过长函数、深层嵌套） |
| 架构设计 | 模块划分合理性、依赖关系、接口设计一致性（统一 BaseCrudController/CrudDialog 模式）、渐进式三层架构 API 管理框架的应用情况 |
| 功能实现 | 功能完整性、边界处理 |
| 性能 | 响应速度、资源占用、并发能力 |
| 可维护性 | 文档完整性、测试覆盖率、可读性 |
| 兼容性 | 浏览器兼容、设备适配 |
| 可复用性 | 重复业务逻辑、分散配置参数、可复用 UI 组件、基础服务（DB 连接、API 请求封装、缓存、日志等） |
| 冗余清理 | 未用代码/文件/表/字段/DTO/依赖/路由等（见 5.2 全部） |

### 6.2 执行流程

```
全面扫描 → 生成报告 → 逐项修复 → 逐项核对 → 全部通过 → 删除报告
```

**阶段一：全面扫描**

- 按 6.1 所有维度逐类扫描，记录位置、重复次数、影响范围
- 使用 Playwright skill 进行全页面遍历，辅助发现前端页面级问题

**阶段二：生成报告**

- 生成 `AUDIT-REPORT.md` 到项目根目录
- 包含：审查概要、问题清单（编号/类型/位置/描述/方案/状态）、修复进度表

**阶段三：逐项修复**

- 按报告编号顺序逐项修复，全程遵守第二章至第五章规范
- 修复后更新报告状态为 ✅

**阶段四：逐项核对**

- 逐项检查 ✅ 项确认修复到位，检查 ⬜ 项确认无遗漏
- 执行构建验证，确认未引入新问题
- 循环直到所有项均为 ✅ 且构建通过

**阶段五：确认完成**

- 全部通过后，删除 `AUDIT-REPORT.md`
- 按 [4.3 格式](#43-必须写入-troubleshootingmd) 将本次审查修复概要记录到 `TROUBLESHOOTING.md`

## 七、安全审查

> **触发时机**：架构审查时同步执行，或用户单独要求。

### 7.1 安全检查要点

按风险等级列出问题，给出整改方案。

| 风险领域 | 检查项 |
|----------|--------|
| 认证与鉴权 | 权限边界、多模块统一鉴权机制、越权风险 |
| API 安全 | 未授权访问、参数注入、越权调用、敏感信息泄露 |
| 多 Agent 安全 | 通信安全、调度指令防注入、Agent 权限隔离 |
| 数据安全 | 向量库/PostgreSQL 访问控制、SQL 注入、数据脱敏、审计日志 |
| 数据流转 | 多库多表同步是否存在泄露、越权读取、脏数据 |
| 前端安全 | XSS、CSRF、敏感信息明文存储 |
| 整体架构安全 | 权限冗余/缺失、敏感数据未加密、日志审计缺失、链路安全、接口限流/防刷、配置泄露、密钥硬编码 |

### 7.2 审查流程

1. 逐项检查，形成安全风险清单（风险等级 + 问题描述 + 整改方案）
2. 按紧急程度逐项修复，修复后复测
3. 将安全修复记录写入 `TROUBLESHOOTING.md`

## 八、测试规范

### 8.1 测试层次与策略

严格执行以下顺序：

| 层次 | 内容 |
|------|------|
| 测试计划 | 分析需求 → 设计测试方案 → 制定测试策略和用例 |
| API 测试 | 覆盖所有接口（含 REST 测试），验证功能、参数校验、异常处理 |
| E2E 测试 | 使用 Playwright 执行端到端测试，模拟真实用户流程 |
| 浏览器 QA | UI 验证 + 冒烟测试，确保页面渲染、交互、兼容性正常 |

### 8.2 Playwright 全页面遍历

- 使用 Playwright skill 对项目所有页面进行自动化遍历
- 检查：崩溃、控制台报错、UI 断裂、加载异常
- 遍历结果纳入测试报告

### 8.3 缺陷管理

- 按严重程度分级，记录复现步骤
- 缺陷修复后必须回归验证，并记录到 `TROUBLESHOOTING.md`

### 8.4 测试闭环

```
测试 → 发现问题 → 修复 → 回归测试 → 通过
```

> 修复过程同样需要遵守第二、三、四章所有规则。

## 九、部署与日志规范

### 9.1 Docker 化部署

测试服务器和生产服务器**统一使用 Docker** 进行部署和验证。

**版本标识：**

- 每次构建镜像必须使用明确的版本号标签（如 `v1.2.3` 或 `YYYYMMDD-HHMM`）
- **禁止使用 `latest` 作为生产部署标签**

**全量清理旧版本：**

- 部署新版本前，必须清除旧的容器、镜像、挂载卷等所有相关资源
- 确保环境干净，避免残留文件或配置污染

**部署验证：**

新版本部署后，必须验证：

- [ ] 容器健康状态（如健康检查接口）
- [ ] 关键接口可达性
- [ ] 主要功能链路通畅
- [ ] 数据持久化完整
- [ ] 部署脚本或命令必须可重复执行，并保留回滚手段

### 9.2 全局日志管理

**日志覆盖范围：**

每个项目必须建立从请求入口到响应输出的**全链路日志**，覆盖所有服务节点、关键功能、性能指标、异常信息。

**日志内容要求：**

| 日志类型 | 内容要求 |
|----------|----------|
| 请求日志 | 时间、来源 IP、请求方法、路径、参数（脱敏后）、响应码、耗时 |
| 业务日志 | 关键操作（增删改）、操作人、操作对象、结果 |
| 错误日志 | 异常堆栈、上下文数据、错误码 |
| 性能日志 | 慢查询、慢接口、资源消耗（CPU/内存） |
| 第三方调用日志 | 外部 API 调用参数、响应、耗时、状态 |

**日志格式：**

- 统一使用结构化格式（如 JSON），方便日志系统收集、检索和分析

**存储与检索：**

- 日志必须集中收集（如 ELK、Loki），支持按 TraceID 串联
- 提供可视化查询界面或命令行工具，确保问题能快速定位

**日志级别：**

- 动态可调整，生产环境默认 INFO
- 问题排查时可按需调整为 DEBUG，但需有自动恢复机制

**审计日志：**

- 涉及安全、数据变更的操作必须落盘，不可被篡改，并满足合规要求

## 十、最终报告与收尾

所有审查（架构、清洁、安全）、测试、部署完成后，生成一份最终效果 MD 文档，概述：

1. 审查覆盖范围
2. 发现问题总数及分类统计
3. 修复完成情况
4. 最终质量评估（功能、性能、安全、可维护性等）

> 确保该文档与 `TROUBLESHOOTING.md` 内容吻合，无遗漏。

完成后删除中间过程报告（如 `AUDIT-REPORT.md`），仅保留最终效果文档和 `TROUBLESHOOTING.md`。

## 附录：统一化标准速查

| 维度 | 要求 |
|------|------|
| 接口设计 | 同类功能统一入参/出参，优先使用 BaseCrudController / CrudDialog 模式 |
| 调用规范 | 前端 composable 用 `use` 前缀，后端 service 用 `Impl` 后缀；统一调用路径 |
| 公共模块管理 | 固定目录，新增需在 index 导出；禁止绕过公共层直接复制 |
| 版本控制 | 公共模块修改需确认所有调用方兼容；破坏性变更需同步更新 |
| 高内聚低耦合 | 公共模块只暴露必要接口，内部实现不可见 |
